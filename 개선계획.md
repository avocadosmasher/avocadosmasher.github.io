# 블로그 개선 계획

앞으로 여유가 생기거나 필요해질 때 적용할 개선 아이디어를 모아두는 문서입니다.
각 항목은 "왜 필요한지 · 어떻게 할지"를 함께 적어, 나중에 바로 착수할 수 있도록 정리합니다.

---

## 1. 블로그 목록 페이징 (카테고리·태그·검색과 호환)

### 배경
`src/pages/blog/index.astro` 는 현재 모든 글을 한 번에 렌더링하고, 사이드바 필터·인라인 검색은 클라이언트 사이드 `display:none` 토글로 처리합니다. 글 개수가 적을 때는 문제 없지만, 늘어나면 다음이 불편해집니다.

- 초기 로드가 무거워짐
- 스크롤이 길어져 원하는 글에 도달하는 시간이 길어짐
- 사용자 관점의 "몇 페이지째"라는 위치 감각이 사라짐

### 목표
**기존 필터(카테고리·태그·인라인 검색)와 완전히 호환되는 페이징**.
즉, 필터로 좁힌 결과 안에서 페이지가 나뉘고, URL이 상태를 그대로 담아야 합니다.

### 접근: 클라이언트 페이징 + URL 상태 동기화

- 카드는 지금처럼 전부 DOM에 렌더링 (Astro 정적 생성 유지)
- `apply()` 안에서 필터 통과 카드 목록을 만든 뒤, 페이지 크기(N개)만큼만 노출
- URL 예시: `?category=Frontend&tag=React&page=2`
  - 필터가 바뀌면 `page=1` 로 초기화
  - 페이지만 바뀌면 필터는 유지
- `history.pushState` 로 URL을 갱신 → 뒤로가기·북마크·공유·글 상세에서 목록으로 복귀까지 자연 복원
- 페이지네이션 UI (이전/다음 + 페이지 번호) 를 카드 그리드 하단에 배치

### 트레이드오프 & 판단 기준
- 초기 HTML에 모든 카드가 실림 → 개인 블로그 규모(수백 편)까지는 문제없음
- 그 이상으로 늘거나 초기 페이로드가 눈에 띄게 커지면 → 페이지 단위로 데이터를 fetch 하는 방식으로 리팩터링
- 검토 기준: **글 100편 근처 도달 시** 도입, 200편 넘어가면 페이로드 방식 재검토

### 대안 (참고용, 채택하지 않음)
- **"더 보기" 버튼**: 구현은 가장 단순하지만 뒤로가기 시 노출 상태가 초기화되어 UX가 나빠짐
- **정적 페이징 (`/blog/page/2/`)**: SEO는 최고지만 클라이언트 필터와의 조율이 까다로움 (페이지 이동 시 필터 상태 유실)

### 참고 위치
- `src/pages/blog/index.astro` — 목록 페이지와 필터 스크립트
- `src/components/PostCard.astro` — 카드 마크업 (data-\* 속성으로 필터링 중)
