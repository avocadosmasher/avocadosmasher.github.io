# devlog 운영 가이드

블로그를 처음 올리는 순간부터, 매일 글을 쓰고 관리하는 일까지의 전체 흐름입니다.

---

## 한눈에 보는 흐름

```
[내 컴퓨터]                          [GitHub]                 [인터넷]
  글 작성(.md)  ──  git push  ──▶  Actions 자동 빌드  ──▶  Pages 에 배포
  npm run dev 로 미리보기
```

- **딱 한 번 하는 일**: 아래 1단계(초기 배포)
- **평소 반복하는 일**: 2단계(글 쓰고 push) — 이게 전부입니다

---

## 1. 초기 배포 (처음 한 번만)

### 1-1. 준비물 확인

```bash
node -v   # v18.20.8 이상 또는 v20+ 이면 OK
```

### 1-2. 프로젝트 설치

압축을 푼 폴더에서:

```bash
npm install      # 최초 1회 (몇 분 소요)
npm run dev      # http://localhost:4321/devlog/ 로 미리보기
```

### 1-3. 내 정보로 바꾸기

- `src/consts.ts` → 이름·소개·이메일·소셜·카테고리·기술 스택
- `astro.config.mjs` → 배포 주소 설정 (아래)

배포 주소는 두 가지 중 하나입니다:

| 방식            | 주소                   | 설정                                               |
| ------------- | -------------------- | ------------------------------------------------ |
| 프로젝트 페이지 (기본) | `유저명.github.io/레포명/` | `site: 'https://유저명.github.io'`, `base: '/레포명/'` |
| 유저 페이지        | `유저명.github.io/`     | 레포 이름을 `유저명.github.io` 로 만들고 `base` 줄 삭제         |

> `base` 를 잘못 맞추면 사이트는 뜨는데 디자인이 전부 깨져 보입니다. 가장 흔한 실수이니 주소 방식을 먼저 정하세요.

### 1-4. GitHub 에 올리기

GitHub 에서 새 레포를 먼저 만든 뒤:

```bash
git init
git add .
git commit -m "init blog"
git branch -M main
git remote add origin https://github.com/유저명/레포명.git
git push -u origin main
```

### 1-5. Pages 켜기

레포 → **Settings → Pages → Build and deployment → Source** 를 **GitHub Actions** 로 선택합니다. (한 번만 하면 됩니다.)

### 1-6. 배포 확인

레포의 **Actions** 탭에서 워크플로가 초록 체크(✓)로 끝나면, 위 표의 주소로 접속해 확인합니다. 처음 한 번은 반영까지 2~3분 걸릴 수 있습니다.

---

## 2. 평소 글쓰기 (가장 자주 하는 일)

### 2-1. 미리보기 서버 켜기 (선택)

```bash
npm run dev
```

글을 저장할 때마다 브라우저가 자동으로 새로고침됩니다.

### 2-2. 새 글 파일 만들기

`src/content/blog/` 안에 `.md` 파일을 만듭니다. **파일 이름이 그대로 주소가 됩니다.**

- `my-first-post.md` → `/blog/my-first-post/`
- 영문 소문자 + 하이픈(-) 권장 (한글 파일명도 되지만 주소가 지저분해집니다)

내용 형식:

```markdown
---
title: 글 제목
description: 카드·검색에 보이는 한 줄 요약
category: Frontend
tags: [React, 성능]
pubDate: 2026-06-30
readingTime: 6분
draft: false
---

여기부터 본문을 마크다운으로 씁니다.

## 이렇게 ## (h2) 와 ### (h3) 로 쓴 제목이
오른쪽 목차에 자동으로 들어갑니다.
```

프론트매터(맨 위 `---` 사이) 필드 정리:

| 필드 | 필수 | 설명 |
|------|:---:|------|
| `title` | ✅ | 글 제목 |
| `description` | ✅ | 카드·검색·SEO 에 쓰이는 요약 |
| `category` | ✅ | `consts.ts` 의 `CATEGORIES` 중 하나여야 함 |
| `tags` | | 태그 배열. 예: `[React, 성능]` |
| `pubDate` | ✅ | 발행일. 최신순 정렬 기준 (`YYYY-MM-DD`) |
| `readingTime` | | 읽는 시간. 비워도 됨 |
| `draft` | | `true` 면 사이트에 안 나옴 (임시저장) |

### 2-3. 올리기

```bash
git add .
git commit -m "post: 글 제목"
git push
```

push 후 **Actions** 탭에서 빌드가 끝나면(1~2분) 자동 반영됩니다.

## 3. 자주 하는 운영 작업

### 글 수정
해당 `.md` 파일을 고치고 다시 `git add → commit → push`. 주소는 그대로 유지됩니다.

### 글 삭제
`.md` 파일을 지우고 push 하면 해당 글과 주소가 사라집니다.

### 임시저장 (초안)
프론트매터에 `draft: true` 를 넣으면 목록·검색·페이지에서 모두 제외됩니다.
완성되면 `draft: false` 로 바꿔 push 하세요.
(로컬에서 미리 보고 싶을 때도 잠깐 `false` 로 두고 `npm run dev` 로 확인하면 됩니다.)

### 새 카테고리 추가
`src/consts.ts` 의 `CATEGORIES` 에 한 줄 추가하면 끝입니다. 사이드바·카드·라벨 색까지 자동 반영됩니다.

```ts
export const CATEGORIES: Record<string, string> = {
  Frontend: '#5849E0',
  Backend: '#0EA5A4',
  DevOps: '#F0883E',
  Database: '#D6336C',
  '회고/생각': '#8B5CF6',
  Security: '#16A34A',   // ← 이렇게 추가
};
```

### 글에 이미지 넣기
이미지 파일을 글과 **같은 폴더**(`src/content/blog/`)에 두고, 상대경로로 참조하는 방식을 권장합니다. base 주소에 상관없이 항상 정상 작동하고, Astro 가 자동 최적화도 해줍니다.

```markdown
![설명 글](./my-diagram.png)
```

> 파비콘처럼 사이트 전역 자산은 `public/` 폴더에 둡니다. 단, `public/` 의 파일을 본문에서 참조할 땐 base 주소를 신경 써야 하므로, 글 이미지는 위의 상대경로 방식이 더 편합니다.

**반드시 위 마크다운 문법(`![]()`)으로만 넣으세요.** 아래 두 방식은 쓰지 않습니다.

- Obsidian 의 `![[파일명.png]]` 위키링크 문법 — 브라우저는 이걸 이미지로 인식하지 못하고 글자 그대로 노출됩니다. Obsidian 에서 쓴 글을 옮길 때 꼭 `![alt](./파일명.png)` 형태로 바꿔주세요.
- `<img src="...">` 같은 raw HTML 태그 — Astro 의 이미지 최적화·복사 파이프라인은 마크다운 `![]()` 문법만 처리합니다. HTML `<img>` 로 넣으면 빌드 시 파일이 `dist/` 에 복사되지 않아 배포 후 이미지가 깨집니다.

**세로로 긴 이미지(모바일 스크린샷 등)** 는 폭 100%로 그대로 두면 너무 커 보일 수 있습니다. 이럴 땐 `<div>` 로 감싸 폭을 제한하세요 — 이때도 안쪽은 반드시 마크다운 문법을 쓰고, 앞뒤로 빈 줄을 넣어야 정상 처리됩니다.

```markdown
<div style="max-width: 40%; margin: 0 auto;">

![설명 글](./mobile-screenshot.png)

</div>
```

### 디자인 손보기
포인트 색만 바꾸려면 `src/styles/global.css` 맨 위 `:root` 의 `--accent` 한 줄만 수정하면 전체에 반영됩니다.

## 4. 문제가 생겼을 때

### Actions 가 빨간불(빌드 실패)
대부분 프론트매터 문제입니다: 필수 필드 누락, `category` 가 `CATEGORIES` 에 없는 값, `pubDate` 형식 오류 등.

> **습관 팁**: push 전에 로컬에서 한 번 `npm run build` 를 돌려보세요. 똑같은 에러를 미리 잡을 수 있어 헛걸음을 막아줍니다.

### 사이트는 뜨는데 디자인이 다 깨짐
`astro.config.mjs` 의 `base` 설정이 실제 주소와 안 맞는 경우입니다. (프로젝트 페이지인데 `base` 가 빠졌거나, 유저 페이지인데 `base` 가 남아있음.) 1-3 의 표를 다시 확인하세요.

### 이미지가 안 보임
경로 문제일 가능성이 큽니다. 3장의 상대경로(`./파일명`) 방식으로 바꿔보세요.

### 변경이 반영이 안 됨
순서대로 확인: ① `git push` 가 실제로 됐는지 ② **Actions** 탭에서 빌드가 아직 도는 중인지(또는 실패했는지) ③ 브라우저 새로고침(캐시) — `Cmd/Ctrl + Shift + R`.

### 글씨체가 평범하게 보임
구글 폰트가 차단된 네트워크(일부 사내망 등)일 때 생깁니다. 일반 방문자에게는 정상적으로 보이니 걱정하지 않아도 됩니다.

---

## 5. 백업과 안전

- GitHub 레포 자체가 백업입니다. 모든 변경 이력이 `main` 브랜치에 남습니다.
- 실수로 글을 지웠어도 git 히스토리에서 되살릴 수 있습니다.
- 큰 개편을 할 땐 새 브랜치를 만들어 작업하고, 잘 되면 `main` 에 합치는 방식이 안전합니다. (선택)
